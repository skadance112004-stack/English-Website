H1. Teacher portal routes are not protected, and role is not checked
Evidence:
app/routes.ts:7-14 exposes /dashboard, /settings, /courses, /courses/create, and all editor routes directly.
app/auth/ProtectedRoute.tsx:16-20 has an auth redirect, but the route table does not use it.
app/auth/ProtectedRoute.tsx:16-20 only checks user, not users/{uid}.role.
app/models/teacherModel.ts:77-83 creates teacher profiles with role: "teacher".
app/routes/dashboard.tsx:100-103 returns early when !user, leaving the screen in a loading/no-op state.
Impact: Unauthenticated users can open teacher URLs, and any authenticated Firebase account can reach teacher UI if Firestore rules do not stop later actions. Pages then hang, show partial UI, or silently no-op.
Suggested fix: Wrap all teacher routes in a protected layout or loader that waits for auth, fetches the user profile/custom claims, requires role === "teacher", and redirects unauthenticated or unauthorized users.
H2. First-time provider login from the login page skips teacher profile creation
Evidence:
app/auth/Login/Login.tsx:33-49 calls signInWithPop@up for Google/Facebook and then navigates to /dashboard.
app/firebase/AuthService.ts:35-49 creates a Firestore teacher profile only in the signup Google path.
app/routes/settings.tsx:105-106 refuses to save if profile is missing.
Impact: A new teacher who clicks Google or Facebook on the login screen gets an Auth user without users/{uid} or teacher stats. Backend AI calls then fail verifyTeacher, and settings cannot save.
Suggested fix: Route all provider sign-ins through one auth service that calls an ensureTeacherProfile() flow before navigation.
H3. Callable TTS lets any teacher write audio into any course
Evidence:
functions/src/generateTTSAudio.ts:54-59 checks only that the caller's user document has role teacher.
functions/src/generateTTSAudio.ts:112-116 accepts arbitrary courseId and exerciseId.
functions/src/generateTTSAudio.ts:196-200 writes with Admin SDK to courses/{courseId}/exercises/{exerciseId}/lines/{lineId}.
Impact: Any teacher account can create or overwrite audio URLs in another teacher's course by passing another courseId.
Suggested fix: Before generating or writing audio, load courses/{courseId} and require createdBy or instructor.id to equal request.auth.uid. Verify the target exercise and line belong to that course.
H4. Video processing trusts Storage path input and publishes HLS outputs publicly
Evidence:
functions/src/processVideo.ts:11-14 triggers on any finalized video object.
functions/src/processVideo.ts:23-39 derives courseId, lessonId, and blockId from the object path.
functions/src/processVideo.ts:90-96 calls makePublic() and uses public URLs for HLS files.
functions/src/processVideo.ts:102-104 updates the lesson block with Admin SDK.
Impact: If Storage rules permit a crafted upload path, the function can update a lesson block derived from that path. Processed lesson videos are also made public outside Firebase authorization.
Suggested fix: Validate uploader/course ownership from Firestore or object metadata before processing. Store HLS outputs behind signed/tokenized URLs instead of makePublic().
H5. Firebase deploy configuration is incomplete or points at the wrong output
Evidence:
firebase.json:5-6 deploys Hosting from public, but the app builds to build/client and build/server.
react-router.config.ts:6 enables SSR.
firebase.json:13-18 references firestore.rules, firestore.indexes.json, and storage.rules, but those files are not present.
Impact: firebase deploy cannot reliably deploy rules/storage, and hosting deploys only static files from public instead of the actual app or SSR entry.
Suggested fix: Add the referenced rules/index files and configure hosting for the real app output/SSR target. If the promo site is deployed separately, add a separate Firebase hosting target for promo-site/dist.
H6. Course list reads a non-schema user mirror that can drift from root courses
Evidence:
GEMINI.md:4-34 defines canonical courses at courses/{courseId} with createdBy.
app/models/courseModel.ts:97-104 reads users/{uid}/courses.
app/models/courseModel.ts:123-128 writes both root courses/{id} and users/{uid}/courses/{id}.
Impact: Courses created or updated according to the documented root schema can be missing from dashboard/course management. The root document and user mirror can diverge.
Suggested fix: Query courses with where("createdBy", "==", uid) and sort by updatedAt. If mirrors are retained, maintain them transactionally/server-side and document them.
H7. Managing an existing course can overwrite aggregates with empty state
Evidence:
app/routes/CoursesManagement.tsx:255-257 navigates to edit with only { courseId, mode: "edit" }.
app/routes/courseInfo.tsx:39-61 fetches the course into form state.
app/routes/courseInfo.tsx:114-158 still derives sections, totals, ratings, and firstFiveLessons from the original existingData, not the fetched course or subcollections.
Impact: Saving or publishing an existing course can reset totalLessons, totalExercises, firstFiveLessons, ratings, and related fields to empty/zero values.
Suggested fix: Store the fetched course and sections in component state, and use that canonical state when saving. Recompute aggregates from actual section/lesson/exercise subcollections.
H8. Course lesson list relies on embedded section.items instead of normalized lesson/exercise docs
Evidence:
GEMINI.md:41-49, GEMINI.md:52-67, and GEMINI.md:138-154 define separate sections, lessons, and exercises subcollections.
app/models/courseModel.ts:168-177 stores items inside section documents.
app/models/courseModel.ts:199-212 returns only data.items || [].
Impact: A course following the documented schema can have real lessons/exercises in subcollections, but the Course Lessons screen will show empty sections.
Suggested fix: Load sections, lessons, and exercises separately; group items by sectionId; keep section docs lightweight unless the schema is intentionally changed.
H9. Editor URLs are not reload or deep-link safe

H10. Lesson saving can report success without saving lesson metadata
Evidence:
app/routes/LessonBuilder.tsx:1607-1629 only saves lesson data inside if (stateData?.courseInfo).
app/routes/LessonBuilder.tsx:1620 calls saveLesson(...) inside a map without await.
app/routes/LessonBuilder.tsx:1623-1631 saves blocks/sections and navigates while the lesson metadata write may still be pending or failed.
Impact: Direct-loaded editors may save only blocks or nothing. Save & Close can return before lesson metadata is written, and failures are unhandled.
Suggested fix: Compute section updates separately, always await saveLesson when courseId/lessonId are known, then save blocks/sections and navigate only after all required writes finish.

H14. Video transcoding races with block persistence
Evidence:
